# 企业内部使用风险判断框架

这是保守预警框架，不是法律定性表。

## 常见结论倾向

- `MIT / BSD / Apache-2.0`
  通常可判为 `低风险`，前提是未发现额外商用限制，且用户确为企业内部使用。

- `GPL-2.0 / GPL-3.0`
  在纯内部使用、无对外分发前提下，通常不直接等于禁止使用；但若用户描述混入分发、嵌入产品、交付客户等因素，不再属于本 Skill 边界。
  对纯内部使用可先给 `信息不足` 或偏保守说明，再建议法务复核具体使用链路。

- `LGPL`
  仅内部使用时通常不直接等于高风险，但若涉及静态链接、深度修改、再打包，容易越出本 Skill 边界。
  若用户只说“公司内部直接安装使用”，通常可在保守说明后给 `信息不足` 或较低风险倾向。

- `AGPL / SSPL / BSL / Commons Clause`
  即使用户当前声明为内部使用，也要特别谨慎。
  若官方文本、FAQ 或商业条款存在明显限制，优先给 `高风险` 或 `信息不足`，不要轻易给 `低风险`。

- `商业闭源 / 商业试用 / Freemium`
  若官方明确要求企业购买商业授权，直接给 `高风险`。
  若存在免费层但企业使用范围不清晰，给 `信息不足`。

- `双许可证`
  必须说明当前版本默认落在哪个许可证上，以及商业授权是否是可选项还是必选项。
  未查清前默认 `信息不足`。

## 保守输出规则

- 能证明“没问题”之前，不要写成“安全”
- 一旦版本未知、项目同名或来源冲突，优先 `信息不足`
- 只有当官方文本与版本范围都比较清晰时，才给 `低风险`
