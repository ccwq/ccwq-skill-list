# 软件内部使用授权合规预警 Skill

## 目标

面向 **企业内部使用** 场景，基于公开信息做一次 **偏保守的法务预警**，帮助用户快速识别：

- 是否明显可用于企业内部使用
- 是否存在需要法务重点复核的限制项
- 是否因为版本、条款或信息不足而无法安全下结论

仅输出合规预警，不输出法律定性意见。

若项目存在商业授权、企业版、订阅计划、试用期、席位限制、按量计费或其他潜在付费要求，还要补充一份面向技术负责人和采购的“授权与付费整理”。

## 边界

仅处理以下场景：

- 公司内部安装和运行软件
- 内部研发、测试、办公、数据处理、运维辅助
- 内部系统或内部工具使用，但 **不对外提供服务**

明确不覆盖以下场景；若用户问题实际属于这些场景，先说明超出本 Skill 边界，再请用户改用更合适的分析方式：

- SaaS / 托管服务 / API 对外提供
- 将软件嵌入自家产品后对外分发
- OEM、转售、白标、商业再许可
- 因分发、联网服务或源码修改而触发的外部开源义务分析

默认按 **中国团队采购视角** 写结论，优先整理中国团队通常能直接看到的购买路径、币种和是否需要联系销售。

## 处理流程

### 1. 先收集稳定事实，再决定是否追问

优先基于用户给出的名称、链接、路径或包名识别项目，并先收集以下稳定事实：

- 官方项目名称
- 官方仓库或官网
- 可确认的许可证文本或官方许可说明
- 是否存在版本切换、双许可证或商业版
- 是否存在公开 pricing、sales、billing、plan 页面

仅在 **缺口会直接影响内部使用结论** 时再追问，优先问这些问题：

- `您说的是哪个项目？` 仅当同名项目不止一个时提问
- `您计划使用哪个版本？` 仅当许可证存在版本差异时提问
- `是否仅限公司内部使用，不会对外提供服务或分发？` 仅当用户描述含糊时提问
- `是否需要修改源码，或只是直接安装使用？` 仅当许可证条款明显依赖修改行为时提问

不要为了形式完整而一次性追问所有问题。能先查明的先查明。

### 2. 按证据优先级调查

优先读取和引用官方来源。证据优先级与冲突处理规则见 [references/source-priority.md](references/source-priority.md)。
付费方式、公开价格、折扣与未知项的整理规则见 [references/pricing-evidence.md](references/pricing-evidence.md)。

至少覆盖以下核查项：

- 当前版本或目标版本的许可证类型
- 官方是否明确允许企业内部使用
- 是否存在商业版 / 企业版 / 双许可证
- 是否发生过许可证变更，且是否影响用户目标版本
- 是否存在附加限制条款、例外条款或单独商业授权要求
- 是否存在公开可见的付费路径，例如月付、年付、按量、按席位、按设备、一次性授权、企业询价
- 是否存在公开折扣，例如年付优惠、教育/非营利折扣、限时活动
- 对于“联系销售”但未公开报价的情况，是否能明确整理为未知项而非猜测价格

常见许可证与内部使用风险判断框架见 [references/internal-use-risk-rubric.md](references/internal-use-risk-rubric.md)。

### 3. 用保守口径给出预警结论

结论使用以下三级：

- `低风险`：当前证据显示，目标版本在企业内部使用下通常可行，未发现明显附加限制
- `高风险`：存在明确限制、商业授权要求、条款冲突，或官方信息显示内部商用需额外授权
- `信息不足`：版本不明、来源冲突、条款模糊，或仅能找到低可信度信息，无法保守地下安全结论

这不是法律意见。若证据不足，不要勉强下 `低风险`。

### 4. 何时必须整理“授权与付费”

只要出现以下任一情况，就必须在主报告之外补充“授权与付费整理”：

- 官方明确要求购买商业授权
- 存在企业版、商业版、Pro/Team/Business/Enterprise 计划
- 存在月付、年付、按量、按席位、按设备、按实例等公开计费方式
- 存在试用期、免费层限制、商用限制或超量收费
- 存在公开折扣信息，例如年付优惠、教育/非营利折扣、限时促销
- 明确写了“联系销售”“custom quote”“enterprise pricing”，即使价格未公开

如果完全未发现额外付费要求，也要用一句话明确写“当前未发现公开的额外付费要求”，不要留空。

## 输出要求

输出 Markdown，采用“双产物”结构：

- `主报告`：继续输出企业内部使用授权合规预警
- `授权与付费整理`：单独附录，面向技术负责人和采购，整理公开付费方式、折扣和未知项

主报告结构如下：

```md
# 软件内部使用授权合规预警报告

## 基本信息
- **软件名称**：
- **目标版本**：如未知，明确写“未知”
- **官方仓库 / 官网**：
- **使用范围**：企业内部使用
- **报告生成时间**：YYYY-MM-DD

## 预警结论

> **低风险 / 高风险 / 信息不足**

一句话说明结论，并明确是“企业内部使用”前提下的判断。

## 大白话简明结论

用 3-5 条短句，直接回答普通业务/采购/老板最关心的问题：

- `能不能用`：先说能不能在公司内部用，避免兜圈子
- `前提是什么`：例如是否必须购买商业授权、是否仅限试用期、是否必须锁定某个版本
- `主要风险`：只写最关键的 1-3 个风险点，例如“长期未授权使用风险高”“版本升级后许可证变化”“实际场景超出内部使用边界”

写法要求：

- 必须用大白话，少用法律术语
- 每条尽量控制在 1 句话内
- 不要把 `信息不足` 写成“应该没问题”
- 如果结论是 `高风险`，第一条就直接写清“不能默认免费商用”或“需要先买授权”这类结论
- 如果结论是 `低风险`，也要顺手补一句“前提是仅限企业内部使用/不对外分发”
```
